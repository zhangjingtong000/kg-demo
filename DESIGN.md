# 3D 知识图谱可视化 — 设计分析

## 一、两个 Demo 的现状描述

### 我们的 Demo（kg-demo/index3d.html）

**视觉风格**
- 深色空间背景（#080c12），带雾效和 600 个背景星点
- 节点：自定义 liquid glass shader，半透明球体 + 菲涅尔边缘光 + 双锐利高光 + 外围 Fresnel 光晕
- 连线：自定义管状几何体，两端略细（78%）、中间略粗（100%），双层叠加——主光束 + 芯光束，都是 AdditiveBlending 发光
- 颜色：每个节点有独立颜色（暖金/灰蓝/淡紫/鼠尾草绿等），去饱和 + 提亮后作为玻璃色
- CSS2D 文字标签悬浮在节点上方，始终面向屏幕

**数据规模**
- 26 个节点，55 条边
- 节点大小 r: 0.48–0.66，分布在 26×20×18 的 3D 空间内
- 数据来源：exercises-dataset 中的健身动作

**交互模型（当前）**
| 操作 | 行为 |
|------|------|
| 拖空白区域 | OrbitControls 旋转/缩放/平移 |
| 鼠标停在节点上 150ms | 高亮该节点 + 直连邻居的线，其他线 dim |
| 点击节点 | 重算径向布局（BFS 距离 → 同心环），节点动画移动，聚焦节点放大到 1.45x + 呼吸动画 |
| 点击空白 | 恢复原始布局 |
| 拖拽节点 | 节点跟随鼠标 + 邻居跟随（衰减 0.35^距离），实时更新连线位置 |

**技术实现**
- Three.js 0.160 + OrbitControls + CSS2DRenderer
- 连接线预建（edgeObjects 数组），拖拽/布局变化时调用 refreshEdgePositions() 更新顶点 buffer，不重建几何体
- 高亮切换只改 material.opacity（setEdgeHighlight），不重建几何体
- 节点位置通过 targets 数组 + render loop lerp 驱动动画

### 3D Force Graph Demo（vasturiano/3d-force-graph）

**视觉风格**
- 无背景星点，纯色背景（通常深色）
- 节点：微小球体（r 极小），可选自定义几何体、图片节点、HTML 节点
- 连线：细线，支持曲线/直线/粒子流动/方向箭头/渐变
- 支持 bloom 后处理发光
- 文字标签通过 CSS/HTML/Sprite 实现

**数据规模**
- 示例中从几十到几百节点不等
- 节点分布在球形/随机空间中
- 力导向布局：d3-force-3d 或 ngraph 物理引擎

**交互模型**
| 操作 | 行为 |
|------|------|
| 拖空白区域 | OrbitControls（或 FlyControls）旋转/缩放 |
| 悬浮节点 | 高亮该节点及其直接连接的边，其他节点/边 dim |
| 点击节点 | 可选：聚焦（镜头拉近 + 高亮子树）、展开/折叠子树 |
| 拖动节点 | 力模拟实时响应——整个图结构弹性变形，松手后力模拟继续运行 |
| 无点击空白 | 通常无操作，或重置高亮 |

**技术实现**
- Three.js + d3-force-3d（力导向布局引擎）
- 力模拟持续运行：每个节点有速度/加速度，被拖拽时受力，其他节点通过边弹簧力联动
- 渲染：每帧根据力模拟的最新位置渲染节点和连线
- 高亮：通过改变节点/边的颜色和透明度实现

## 二、核心差异对比

| 维度 | 我们 | 3D Force Graph |
|------|------|---------------|
| **布局引擎** | 硬编码坐标 + 点击时 BFS 径向重排 + lerp 动画 | d3-force-3d 持续力模拟 |
| **节点视觉** | 大（r 0.48-0.66），半透明玻璃质感，自定义 shader | 小，不透明，风格简洁 |
| **连线视觉** | 发光管状，端细中粗，双层叠加 | 细线，可选粒子/箭头/渐变 |
| **拖动效果** | 仅单节点移动 + 邻居跟随衰减 | 力模拟全局响应，弹性变形 |
| **聚焦** | 全图重排为径向同心环 | 镜头拉近 + 高亮子树 |
| **复杂度处理** | 全部显示，靠 hover 高亮区分 | 节点小、线细 + hover 高亮 |
| **性能策略** | 预建几何体，拖拽时更新顶点 buffer | 力模拟每帧更新，渲染跟随 |
| **自由度** | 节点可以被拖到任意位置，布局可被手动打乱 | 力模拟始终在运行，节点无法被永久拖离 |

## 三、可借鉴与不该借鉴

### 应该借鉴的

**1. 力模拟（部分借鉴）**
- 原因：解决「拖动只有单节点动」的机械感
- 借鉴方式：不一定要完整的 d3-force。可以只加一个轻量的弹簧力——每条边像一个弹簧，节点被拖远时弹簧拉回，邻居被带动。这比「衰减推动邻居」更自然
- 潜在冲突：力模拟和我们的「径向布局」是两个不同的位置控制逻辑。力模拟持续运行，径向布局是一次性设置 targets，两者会打架
- 冲突解决：设定两个模式——「自由探索模式」用力模拟，「聚焦模式」用径向布局。两者互斥，切换时做过渡动画

**2. 悬浮高亮的交互节奏**
- 原因：3d-force-graph 的高亮反应非常快且干净——只亮直连的边和节点
- 借鉴方式：保持我们现在的 debounce 150ms，但高亮范围只限于直连边（我们已经做到了）
- 无需大改

**3. 大量节点时的视觉策略**
- 原因：3d-force-graph 节点小、连线细，信息密度高但不乱
- 借鉴方式：我们的节点大、连线发光——适合 20-40 个节点的精品展示，不适合 100+ 节点。如果未来要扩展节点数，要么缩小节点（失去玻璃质感），要么做分层/聚类（LOD）
- 无需现在改，但要知道这个限制

### 不应该借鉴的

**1. 微小球体节点**
- 原因：我们的视觉差异化和高级感正是来自 liquid glass 大球体 + 光晕。换成小球就失去了这个特色
- 替代方案：保持现有节点风格，通过 hover 高亮解决复杂度问题

**2. 力模拟作为唯一布局**
- 原因：力模拟的布局结果不可预测——有时候乱成一团，有时候很漂亮。我们的径向布局保证了「聚焦时一定是好看的同心圆结构」
- 替代方案：自由探索时可用轻量力模拟，聚焦时用径向布局。两种模式互补

**3. 粒子流动/方向箭头/渐变连线**
- 原因：我们的连线已经有两层发光 + 端细中粗的变化，再加粒子会让视觉过载
- 替代方案：保持现有连线风格，这是我们的差异化

## 四、当前 Demo 已知问题清单

1. **拖动 + 点击冲突**：拖动节点时 mousedown/mouseup 可能触发 click 导致聚焦。已通过 didDrag 标记解决，但需要验证边缘情况
2. **拖动 + OrbitControls 冲突**：拖空白旋转后 mouseup 可能触发 unfocus click。已通过 mouseDownPos 距离判断解决
3. **拖动邻居跟随不自然**：衰减系数 0.35 是拍脑袋的，没有物理依据
4. **没有力回弹**：节点被拖走后不会自动弹回，必须手动点空白恢复
5. **性能**：预建几何体更新顶点 buffer 的方案可行，但 refreshEdgePositions 在 55 条边 × 每边 2 层 × 每层 ~300 顶点的场景下仍有一定开销
6. **节点在半透明状态下看不清**：玻璃材质在深色背景上通透感不足
7. **label 重叠**：多节点时 CSS label 会互相遮挡，无避让逻辑
