# 文档 → 知识图谱 抽取管道设计

## 整体流程

```
PDF/Markdown/文本文件
        │
        ▼
┌───────────────────────────────────────────────────┐
│ 步骤 1：文件解析（文字 + 图片分离）                   │
├───────────────────────────────────────────────────┤
│ 借：docling-graph 的 PDF 解析策略                    │
│     MinerU / PyMuPDF 做文字提取 + 图片导出            │
│ 冲突风险：无。解析层独立，不依赖后续步骤               │
└──────────────────────┬────────────────────────────┘
                       │
                       ▼
┌───────────────────────────────────────────────────┐
│ 步骤 2：分块（长文本切割）                            │
├───────────────────────────────────────────────────┤
│ 借：docling-graph 的结构感知分块 + SLIDE 的滑动窗口    │
│     简单文档 → 结构感知 + 30% 重叠                    │
│     复杂文档 → 骨架先行（两轮 LLM 调用）               │
│ 冲突风险：分块策略影响后续实体关系质量。                │
│          块太小 → 丢关系；块太大 → LLM 慢/超时         │
│ 解决方案：每块 300-500 字，30% 重叠，自适应            │
└──────────────────────┬────────────────────────────┘
                       │
                       ▼
┌───────────────────────────────────────────────────┐
│ 步骤 3：内容理解（文字+图片→结构化描述）               │
├───────────────────────────────────────────────────┤
│ 文字 → LLM 直接抽取                                  │
│ 图片/图表 → MLLM 描述后再抽取（仅当图片含关键信息时）    │
│ 借：sift-kg 的多格式处理思路                          │
│ 冲突风险：MLLM 调用成本高、速度慢                      │
│          图片描述质量影响后续抽取                       │
│ 解决方案：普通图片跳过，仅流程图/表格/关系图走 MLLM      │
│          用本地 qwen3-vl:4b 降低成本                  │
└──────────────────────┬────────────────────────────┘
                       │
                       ▼
┌───────────────────────────────────────────────────┐
│ 步骤 4：实体关系抽取（LLM 输出结构化数据）             │
├───────────────────────────────────────────────────┤
│ 借：LightRAG 的分隔符格式 + HiRAG 的两阶段策略        │
│     阶段 1：entity<|>Name<|>Type<|>Desc             │
│     阶段 2：relation<|>Src<|>Tgt<|>Type<|>Desc      │
│     类型约束：预定义 EntityType / RelationType        │
│ 冲突风险：LLM 不严格遵循分隔符格式 → 解析失败           │
│          不同模型输出格式不同（有的加前缀有的不加）      │
│ 解决方案：解析器同时兼容有无前缀两种格式 ✓（已实现）     │
│          提示词中强调输出格式                          │
└──────────────────────┬────────────────────────────┘
                       │
                       ▼
┌───────────────────────────────────────────────────┐
│ 步骤 5：合并去重（多 chunk 结果整合）                  │
├───────────────────────────────────────────────────┤
│ 借：MS GraphRAG 的精确名称匹配合并                     │
│     DEG-RAG 的类型感知去重 + LLM 判定疑似重复           │
│     阶段 1：同名 → 直接合并（保留描述更详细的版本）      │
│     阶段 2：同类型 + 名称相似 > 80% → 合并             │
│     阶段 3：跨块关系合并（同 src/tgt/type 的去重）      │
│ 冲突风险：错误合并不同实体 → 图谱信息丢失               │
│          错误保留重复实体 → 图谱冗余                    │
│ 解决方案：只在同类型内做模糊匹配，不跨类型合并           │
│          模糊匹配备注来源，方便人工审核                 │
└──────────────────────┬────────────────────────────┘
                       │
                       ▼
┌───────────────────────────────────────────────────┐
│ 步骤 6：图存储 + 节点坐标生成                         │
├───────────────────────────────────────────────────┤
│ 存储：NetworkX（轻量）或 Neo4j（完整）                │
│ 坐标：NetworkX spring_layout 生成初始坐标             │
│       或用我们已有的 BFS 径向布局                      │
│ 借：无。这层是纯工程，不需要借。                       │
│ 冲突风险：spring_layout 结果可能重叠/不美观            │
│ 解决方案：用我们的 BFS 径向布局覆盖初始坐标             │
└──────────────────────┬────────────────────────────┘
                       │
                       ▼
┌───────────────────────────────────────────────────┐
│ 步骤 7：API 层（FastAPI）                            │
├───────────────────────────────────────────────────┤
│ 借：AI_KnowledgeGraph_Extractor 的接口设计            │
│     POST /upload  → 上传文件，返回任务 ID             │
│     GET  /graph/{id} → 返回 {nodes, edges}          │
│     POST /chat      → KG + RAG 问答                  │
│ 冲突风险：无。CRUD 接口标准模式。                      │
└──────────────────────┬────────────────────────────┘
                       │
                       ▼
┌───────────────────────────────────────────────────┐
│ 步骤 8：前端展示（3D 图谱 + 聊天面板）                │
├───────────────────────────────────────────────────┤
│ 已有：index3d-bounce.html（我们自己的 3D 可视化）     │
│ 新增：上传页面 + 处理进度 + 侧边聊天框                 │
│ 借：无。展示层是我们的差异化。                         │
└───────────────────────────────────────────────────┘
```

## 业界方案对比（2025）

### 步骤 1：PDF 解析

| 工具 | 优点 | 缺点 | 推荐场景 |
|------|------|------|---------|
| **MinerU** | 中文最强、版面对齐精准、OmniDocBench 开源第一 | AGPL 协议、需 GPU、安装复杂 | 中文学术/财报 |
| **Docling** | MIT 协议、TableFormer 表格识别独一档、多格式支持 | 速度较慢、版面分析不如 MinerU | 企业多格式文档 |
| **PyMuPDF** | 极快（毫秒级）、轻量 | 版面理解有限、扫描件差 | 简单 PDF 预处理 |
| **Marker** | 表格还原最准、速度比 MinerU 快 1 倍 | GPL 协议 | PDF→MD 专项 |

**推荐**：组合管道，各取所长：

```
PDF 输入
  │
  ├─ 版面分析 → MinerU（中文最强，段落/标题/公式识别）
  ├─ 表格提取 → Docling（TableFormer，无线条表格也能识别）
  ├─ 文字提取 → PyMuPDF（极快，简单页面直接过）
  └─ 图片导出 → 所有工具都支持，导出后交给 MLLM
```

优先级：MinerU 主解析 → 遇到复杂表格回退 Docling → 简单页面用 PyMuPDF 加速

### 步骤 2：分块

**2025 年结论**：小块（300-500 tokens）的实体/关系召回率**显著优于**大块（1200+ tokens）。实测数据：

| 块大小 | 平均节点数 | 平均边数 |
|--------|----------|---------|
| 300 tokens | ~100-129 | ~57-120 |
| 1200 tokens | ~9-25 | ~4-20 |

**SLIDE 论文**（2025.3）：重叠滑动窗口 → 实体 +24%、关系 +39%。

**推荐**：每块 **300-500 字**（≈500-800 tokens），**30% 重叠**。简单文本 < 2000 字不分块。

### 步骤 3：内容理解（图片处理）

业界通用做法：**分级处理**

```
图片 → OCR 检测 → 有文字 → MLLM 描述
                → 无文字 → 跳过（装饰图）
```

**推荐**：本地 qwen3-vl:4b 做 OCR + 简单描述（免费），云端 Qwen3-VL-32B 仅处理复杂图表（质量优先时）。

### 步骤 4：实体关系抽取

已验证可行：
- 分隔符格式 `entity<|>Name<|>Type<|>Desc` ✅
- 两阶段策略（先实体、后关系）✅
- 解析器同时兼容有无前缀 ✅
- DeepSeek V4 Pro 速度 21s/短文本 ✅

问题：长文本慢（300s 超时）→ 分块解决。

**推荐**：保持现有提示词设计，加分块后不再有超时问题。

### 步骤 5：合并去重

业界三阶段方案：

| 阶段 | 方法 | 作用 |
|------|------|------|
| 阻塞 | 同类型聚类 + 向量相似度 | 缩小候选范围（O(n²)→O(k·m²)）|
| 匹配 | 精确同名 → 合并；模糊相似 → LLM 判断 | 去重核心 |
| 合并 | 保留描述最详细版本，记录来源 | 避免信息丢失 |

DEG-RAG 论文结论：**40% 实体可安全删除，QA 性能反而提升**。只在**同类型内**做模糊匹配，不跨类型合并。

**推荐**：简化版——同名直接合并 + 同类型 80% 字符重叠合并 + 跨块同边去重。

### 步骤 6：图存储

| 方案 | 优点 | 缺点 |
|------|------|------|
| **NetworkX** | 纯 Python、零配置、内存图 | 不支持持久化、大图慢 |
| **Neo4j** | 专业图数据库、Cypher 查询、可视化、毕设加分 | 需要安装、学习成本 |

**推荐**：Neo4j（你的选择）。

### 步骤 7-8：API + 前端

API 无争议，前端是我们自己的 3D 可视化。

## 需要讨论的决策点

### 决策 1：分块参数

**调研结论**：小块（300-500 tokens）实体/关系召回率比大块高 5-10 倍。SLIDE 论文证实 30% 重叠提升 24% 实体 + 39% 关系。

**推荐**：300-500 字/块、30% 重叠、< 2000 字不分块。直接采用。

### 决策 2：MLLM 调用策略

**调研结论**：通用做法是先 OCR 检测图片有无信息量（有文字/图表才调 MLLM），跳过纯装饰图。

**推荐**：本地 qwen3-vl:4b 做 OCR + 简单描述（免费、快），复杂图表才走云端 32B。如果文档主要是文字为主、图片不多，这步可以放到 P2 后做。

### 决策 3：LLM 模型选择

**最终模型矩阵**：

| | 文字 LLM | 视觉 MLLM |
|---|---|---|
| **云端（主力）** | DeepSeek V4 Pro (硅基流动) | Qwen3-VL-32B (硅基流动) |
| **本地（备用/对比）** | qwen3.5:9b (Ollama) | qwen3-vl:4b (Ollama) |

**实测结论**：本地 qwen3.5:9b（超时），云端 Qwen3-32B（132s/短文本，慢），云端 DeepSeek V4 Pro（21s/短文本，快 6 倍，质量好）。

**视觉策略**：本地 qwen3-vl:4b 做 OCR 初筛（判断图片有无信息量）→ 有信息量送云端 Qwen3-VL-32B 深度理解 → 无信息量跳过。

**推荐**：纯云端 DeepSeek V4 Pro 做文字抽取。保留双后端开关作为论文对比素材。本地后端标注"实验性"。

### 决策 4：存储选型

**你的决定**：Neo4j。
